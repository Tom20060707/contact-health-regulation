# Contact-Health Regulation

这是一份给导师或第一次了解项目的人看的研究说明。它记录的是一个人形机器人强化学习项目：我想用“接触健康状态”去理解和调节 PPO 在复杂接触任务里的训练稳定性。

一句话版本：

> 人形机器人训练崩掉时，reward 往往不是最早报警的信号；脚、手、身体和环境之间的接触模式，可能更早露出问题。这个项目尝试先识别接触状态，再决定 PPO 是该正常学习、提前预警、进入保护，还是在稳定后小心推动探索。

这不是一个只为了包装新缩写的仓库。它更像一份科研工作笔记：前面放已经整理好的项目说明，后面用一个文件保留每日研究记录，方便追溯每个判断是怎么来的。

## 动机故事

PPO 是机器人强化学习里很常用的方法，但在人形机器人上会遇到一个麻烦：机器人不是在空中做动作，它一直在和世界接触。

上楼梯时，脚要踩稳；推物体时，身体、手和物体之间要形成稳定接触；插入任务里，接触精度和阶段切换也很重要。接触模式一乱，策略可能还没立刻掉分，但训练方向已经开始变坏。

这个项目的出发点是：

> 与其等 reward 掉下来以后才发现策略坏了，不如提前看接触状态有没有变得不健康。

这里的“接触健康”不是说动作越整齐越好，而是说：策略和环境之间的接触关系是否还处在一个稳定、可解释、可恢复的范围里。

## 模型和方法

现在的主线叫 **Contact-Health Regulation**，可以理解成一个训练状态调节框架。

它大致分三层：

| 层级 | 作用 | 例子 |
|---|---|---|
| 诊断 | 从接触信号中判断训练状态 | 接触时序、接触面分布、接触稳定分数 |
| 状态判断 | 区分正常、预警、保护、推动、挽救 | normal / warning / protect / hold / push / rescue |
| PPO 调节 | 根据状态改变训练更新方式 | 软门控、保守更新、停止 Push、恢复后再放开 |

早期名字 **TC-PPO** 还会在历史记录里出现，但它已经不是现在的主标题。更准确地说，TC-PPO 是这个项目早期的算法入口；现在的论文主线已经从“一个 PPO 小改动”升级成“接触健康状态如何诊断和调节训练”。

## 实验结果

当前实验更适合被理解成“机制验证”，不是简单的全任务刷榜。

已经比较清楚的部分：

- 在一些支撑接触或接触退化明显的任务里，接触信号能帮助解释训练为什么变坏。
- soft gating 和自适应阈值比早期硬阈值更稳，不容易因为任务天然接触差异而误触发。
- 部分任务上能看到稳定性或方差改善，例如 slide、powerlift 一类任务提供了有价值的证据。
- Mode A/B/C 不是任务好坏排名，而是方法在不同训练状态下的表现边界。

也必须诚实说明的部分：

- 不是所有任务都适合用脚部接触时序解释。
- kitchen、insert、truck 等非周期操作任务需要更细的接触面、任务阶段或物体状态建模。
- Rescue、Push 的一些扩展想法还不能写成已验证结果。
- 实验数字最终要回到论文源码、图表脚本和原始训练日志核对，不能只从讨论摘要里抄。

## 旧名字怎么处理

每日研究日志里会看到这些旧名：

- **TC-PPO**：最早的算法名，用接触时间一致性调节 PPO。
- **FTC / CTS**：早期围绕足部接触和接触时序的诊断指标。
- **CSC**：后来为了覆盖更一般接触面提出的说法。
- **CHR / CHS**：现在更适合对外解释的主线，分别是 Contact-Health Regulation 和 Contact-Health State。

对外介绍时，不建议把 TC-PPO 放在第一位。更好的讲法是：

> 我研究的是 contact-rich humanoid policy optimization 中的接触健康状态调节；TC-PPO 是这个方向早期形成的一个 PPO 稳定化实现。

## 给导师看的最短版本

这个项目关注一个现象：在人形机器人接触任务中，训练失败往往先表现为接触模式退化，而不是 reward 立即崩溃。基于这个观察，我尝试构建 Contact-Health Regulation 框架，用接触状态诊断训练是否健康，并据此调节 PPO 更新。实验目前显示，这个思路在部分接触退化明显的任务上能解释并改善训练稳定性；同时，非周期操作任务也暴露了仅依赖脚部接触时序的边界，提示后续需要更一般的接触面和任务阶段建模。
